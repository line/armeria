# Introduction

## What is a service mesh?

When an application is split into many services, each service needs to find the
others, balance load across their instances, retry failed calls, enforce timeouts,
and encrypt traffic between them. Implementing this logic in every service is
error-prone and hard to keep consistent — especially as the number of services
grows.

A **service mesh** moves these concerns out of application code and into a dedicated
infrastructure layer. It has two parts:

- **Control plane** — a centralized component that watches for service changes
  (new instances, removed endpoints, updated routing rules) and computes the
  configuration that the data plane should enforce.
- **Data plane** — the component that sits on the traffic path, applying the
  configuration it receives: routing requests to the right endpoint, load
  balancing, retrying failures, enforcing timeouts, and encrypting traffic
  with mutual TLS.

The control plane pushes configuration to the data plane using the
**xDS protocol** — originally designed for Envoy but now a
[CNCF standard](https://github.com/cncf/xds) adopted by multiple projects.

## Sidecar vs. proxyless

In a traditional service mesh, the data plane is a **sidecar proxy** — a separate
process (typically Envoy) deployed alongside every application instance. An
orchestrator like Kubernetes injects the sidecar automatically, and all inbound
and outbound traffic is intercepted by it.

This model has clear strengths: it is **language-agnostic** (any process gets
mesh features regardless of its runtime), provides a **strong security
boundary** (policy is enforced by the sidecar even if the application is
compromised), and keeps mesh concerns **completely transparent** to application
code. For organizations already running on Kubernetes with a platform team to
manage the mesh infrastructure, sidecars are a well-proven choice.

That said, some organizations face challenges with this model:

- **Platform dependency** — while sidecars can run outside Kubernetes, the
  operational tooling for injection, traffic interception (via iptables),
  lifecycle management, and health checking is largely Kubernetes-specific.
  Running a sidecar fleet on bare metal or VMs requires significant custom
  automation.
- **Operational overhead** — running a full control plane plus a proxy per
  service instance requires a dedicated team to deploy, upgrade, monitor, and
  troubleshoot the mesh infrastructure alongside the applications.
- **Incremental adoption** — for organizations with large legacy deployments,
  migrating everything to Kubernetes and a service mesh at once is rarely a
  realistic first step.

A **proxyless** approach offers an alternative. Instead of routing traffic
through an external proxy, the application implements the xDS protocol directly
and applies the configuration in-process:

- **No orchestrator required** — service mesh features work on bare metal, VMs,
  and containers alike. There is no dependency on Kubernetes for sidecar injection
  or traffic interception.
- **Simpler deployment** — there is no sidecar fleet to deploy or upgrade.
  The mesh logic is part of the application, versioned and shipped together
  with it.
- **Lower latency** — requests are routed directly to the remote service without
  passing through an intermediate proxy, eliminating two extra network hops.
- **Fewer resources** — no separate proxy process per instance means less memory
  and CPU overhead.
- **Incremental adoption** — you can add xDS support to individual services
  without migrating your entire infrastructure. Services with and without xDS
  can coexist.

### Trade-offs

Proxyless is not a strict upgrade over the sidecar model — it shifts
responsibilities rather than eliminating them:

- **Language-specific** — proxyless only works for languages that have an xDS
  implementation. A polyglot organization may still need sidecars for services
  written in languages without library support.
- **Weaker security boundary** — with sidecars, policy enforcement happens
  outside the application process. In the proxyless model, the application
  itself must be trusted to enforce mTLS and routing policy correctly.
- **Debugging shifts to application teams** — routing and load balancing
  decisions happen inside the application's process rather than in a separate
  proxy with its own admin interface and stats. Application developers need to
  understand xDS semantics when troubleshooting.

The two models are not mutually exclusive — proxyless services and sidecar-based
services can coexist in the same mesh, so you can adopt either approach
incrementally.

## Landscape

There are different approaches to solving the same set of problems in Java:

- [**Spring Cloud**](https://spring.io/projects/spring-cloud) — provides
  client-side service discovery, load balancing, and circuit breaking for Java
  applications using its own ecosystem (Eureka, Consul, Ribbon) rather than the
  xDS protocol.
- [**gRPC xDS**](https://github.com/grpc/proposal/blob/master/A27-xds-global-load-balancing.md)
  — gRPC clients and servers can speak xDS natively to receive routing and load
  balancing configuration from a control plane like Istio.

Armeria's xDS module occupies a similar space — it speaks the xDS protocol and
supports HTTP, gRPC, and Thrift.
See [Concepts](/docs/advanced/xds/concepts) for a deeper look at the architecture.
